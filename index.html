Growth Loops & Flywheels

Self-reinforcing systems where output becomes input

Traditional marketing funnels are linear: acquire, activate, retain. Growth loops are circular: each user's action generates output that feeds back into the system to acquire or activate the next user. A viral loop (user invites friend), a content loop (user creates content that attracts visitors via SEO), or a paid loop (revenue funds ads that acquire users who generate revenue) — each is a self-reinforcing cycle. The best products have multiple interlocking loops that compound over time.

How to apply it

Map every growth mechanism in your product as a loop, not a funnel. Identify the input (new user), the action (what they do), the output (what that action creates), and how the output becomes a new input. Then measure the efficiency of each loop: what percentage of output converts back to input? A viral loop with a K-factor of 0.5 means every 2 users generate 1 new user — not viral on its own, but powerful when compounded with other loops.

Example

LinkedIn's core growth loop: a professional joins → builds their profile → appears in search results and "People You May Know" → their presence attracts former colleagues to join → those colleagues build profiles → the cycle repeats. This loop is supercharged by a content loop: members post articles → articles appear in Google search → visitors discover LinkedIn → they join to engage with the content → they create more content. By 2015, LinkedIn had 400M members largely through these interlocking loops, not paid advertising.

Growth Loop Mapping Template

  1. Input — What triggers the loop? (New user sign-up, content creation, purchase, invitation sent)
  2. Action — What does the user do inside the product? (Complete profile, post content, invite colleagues, engage)
  3. Output — What does that action produce? (SEO content, social proof, invitations, data, revenue)
  4. Reinvestment — How does the output become a new input? (SEO drives sign-ups, invites bring users, revenue funds ads)
  5. Loop Efficiency — What % of output converts back to input? (K-factor for viral, CAC payback for paid, CTR for content)
  6. Compounding Rate — How fast does the loop cycle? (Daily, weekly, monthly? Faster loops compound more.)
  7. Interconnections — How do your loops reinforce each other? (Content loop feeds viral loop feeds paid loop)

North Star Metrics & Metric Hierarchies

One metric to rule them all — and a tree that supports it

A North Star Metric (NSM) captures the core value your product delivers. It's the single metric that, if it grows, means your business is healthy. But a North Star alone isn't enough — you need a metric hierarchy: the NSM at the top, supported by input metrics that teams can directly influence. Without the hierarchy, teams optimize local metrics that may not move the North Star. With it, every team knows exactly how their work connects to company success.

How to apply it

Choose a North Star that reflects value delivered to users, not just revenue extracted. Then decompose it: NSM = Input 1 × Input 2 × Input 3. Each input metric should be owned by a specific team. For example, if your NSM is "Weekly Active Members Who Post," it decomposes into: (new members activated this week) × (returning members) × (posting rate). Growth team owns activation, product team owns retention, content team owns posting rate. Now every team has a clear metric and clear accountability.

Example

LinkedIn's North Star evolved over time. Early on, it was "connections made per week" — because connections drove the network effect. As the platform matured, it shifted to "monthly active sessions" because LinkedIn needed users to come back regularly, not just connect once. The metric hierarchy: MAU = (total members) × (activation rate) × (return frequency). The growth team focused on activation (getting new members to add 5 connections in the first week — a magic number that predicted long-term retention). The feed team focused on return frequency (making the feed valuable enough to check daily).

North Star Metric Framework

  1. Define Core Value — What value does your product deliver? For LinkedIn: "connecting professionals to opportunity." The NSM must measure this.
  2. Choose the NSM — One metric that grows when users get value. Not revenue (that's an output). Not sign-ups (that's an input). The value moment.
  3. Decompose into Inputs — NSM = Input A × Input B × Input C. Each input should be directly influenceable by a team.
  4. Assign Ownership — Each input metric has one team that owns it. No shared ownership — shared ownership is no ownership.
  5. Set Targets — Each input metric gets a quarterly target. If all input targets are hit, the NSM target should follow mathematically.
  6. Counter-Metrics — For each input, define a counter-metric that prevents gaming. (e.g., "connections" countered by "connection acceptance rate")
  7. Review Cadence — NSM reviewed monthly at leadership level. Input metrics reviewed weekly by owning teams.

Cohort Analysis & Retention Curves

Averages lie — cohorts tell the truth about your product

Aggregate metrics hide the story. If your overall retention is 30%, it might mean every cohort retains at 30% (stable) or that old cohorts retain at 50% while new cohorts retain at 10% (crisis). Cohort analysis groups users by when they joined (or when they took an action) and tracks their behaviour over time. Retention curves show the shape of user engagement — the steep initial drop, the flattening point where habits form, and the long-term plateau that determines your product's true health.

How to apply it

Build cohort retention tables weekly: rows are sign-up weeks, columns are weeks since sign-up. Look for three things: (1) Is the retention curve flattening? If it never flattens, users are slowly leaking and your product doesn't create habits. (2) Are newer cohorts retaining better than older ones? This means product improvements are working. (3) Where is the steepest drop? That's where your activation or onboarding is failing. The "magic number" is the activation threshold where retention dramatically improves — find it, then engineer your onboarding to get every user there.

Example

LinkedIn's growth team discovered that users who added 5 connections in their first week had dramatically higher long-term retention than those who added fewer. This "5 connections in 7 days" became LinkedIn's magic number — the activation threshold. The entire onboarding flow was redesigned around this insight: address book import, "People You May Know," company-based suggestions — all engineered to get users to 5 connections as fast as possible. Cohort analysis showed that post-redesign cohorts retained at 2x the rate of pre-redesign cohorts at D30.

Cohort Analysis Playbook

  1. Define the Cohort — Group users by sign-up date, activation date, feature adoption date, or campaign source. The grouping depends on the question.
  2. Choose the Metric — What are you tracking over time? Return visits, purchases, feature usage, revenue? One metric per analysis.
  3. Build the Table — Rows = cohorts. Columns = time periods. Cells = % of cohort still active. Look at it weekly.
  4. Find the Flattening — Where does the retention curve level off? Users past this point are habitual. This is your "core" user base.
  5. Find the Magic Number — What action separates retained users from churned users? (LinkedIn: 5 connections. Facebook: 7 friends in 10 days.)
  6. Compare Cohorts — Are newer cohorts better or worse? If worse, product/market is degrading. If better, your improvements are working.
  7. Segment Within Cohorts — Break cohorts by source, geography, device, or plan. Find which segments retain best and why.

Experimentation & Network Effects

Measuring growth in systems where users affect each other

Standard A/B testing assumes independence — one user's experience doesn't affect another's. But in networked products like LinkedIn, this assumption breaks. If you show User A a new feature that makes them post more, User B sees more content in their feed even though they're in the control group. This "network interference" means standard A/B tests can underestimate (or overestimate) the true effect of changes. Advanced experimentation in networked products requires techniques like cluster-based randomisation and difference-in-differences analysis.

How to apply it

For features that affect only the individual user (e.g., UI changes), standard A/B testing works fine. For features that create network effects (e.g., new ways to connect, content creation tools), use cluster-based randomisation: randomise entire networks or geographic clusters, not individual users. For features you can't A/B test (e.g., pricing changes, platform-wide policies), use pre/post analysis with a control group (difference-in-differences) or synthetic control methods. Always define your primary metric, sample size, and decision criteria before running the test.

Example

LinkedIn's "People You May Know" (PYMK) feature couldn't be properly A/B tested at the individual level because of network effects: if User A gets PYMK suggestions and sends connection requests, User B (in the control group) receives those requests and becomes more engaged. The true effect of PYMK was being leaked into the control group. LinkedIn's data science team used geographic cluster randomisation — entire city-level networks were assigned to treatment or control — to measure the true impact. The result: PYMK's actual effect on connections was 30% higher than the individual-level A/B test suggested, because the test had been underestimating the network amplification effect.

Experimentation Decision Framework

  1. Network Impact Check — Does this feature affect other users? If yes, individual-level A/B testing will give misleading results.
  2. Choose the Method — Individual A/B for isolated features. Cluster randomisation for networked features. Diff-in-diff for untestable changes.
  3. Define Success Criteria — Primary metric, minimum detectable effect, confidence level, sample size — all BEFORE running the test.
  4. Guard Rails — Define metrics that must NOT degrade (e.g., "engagement can't drop >2%"). Stop the test if guardrails are breached.
  5. Run Duration — Long enough for statistical significance AND to capture novelty effects wearing off. Usually 2-4 weeks minimum.
  6. Novelty & Primacy — New features often spike then decay (novelty) or start slow then grow (learning). Wait for the curve to stabilise.
  7. Decision Framework — What happens if results are inconclusive? Ship to 50%? Run longer? Kill it? Decide this before the test ends.
